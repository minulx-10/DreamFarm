# Antigravity 에이전트 작업 지침

이 문서는 사용자의 요청에 따라 에이전트가 지속해서 지켜야 할 규칙을 명시한 문서입니다.

## 📌 규칙 사항
0. **순수 100% 파이썬 (Pure Python Only)**
   - 이 프로젝트는 **오직 파이썬만으로** 구현한다. GitHub 언어 통계에 **Python 100%**로 표시되어야 한다.
   - 다른 프로그래밍 언어 소스(JS/HTML/CSS/C/Cython/셰이더 등)나 컴파일된 확장 모듈을 추가하지 않는다.
     소리는 음원 파일 대신 `numpy` 합성으로, 그래픽도 코드로 그린다.
   - (`.md` 문서·`.yml` 워크플로는 linguist에서 문서/데이터로 분류되어 언어 통계에서 제외되므로 무방하다.)
1. **계획서 및 보고서 한국어 작성**
   - 개발에 필요한 모든 계획서(`implementation_plan.md`), 작업 목록(`task.md`), 작업 완료 보고서(`walkthrough.md`) 등 모든 프로젝트 기획 및 작업 관리 문서는 **반드시 한국어로 작성**해야 합니다.
2. **코드 및 주석 지침**
   - 사용자 피드백을 반영하여 코드 개선 작업을 수행하고, 변경 후에는 정상 동작 여부를 테스트하여 원격 저장소(`main` 브랜치)에 반영합니다.
3. **버전 번호 규칙 (Versioning Rule)**
   - 버전 문자열은 `core/version.py`의 `VERSION` **한 곳에서만** 관리한다 (화면 표시가 여기서 나옴). 배포 시 이 값을 먼저 올린다.
   - **큰 업데이트**(기능 추가·시스템 변경 등)는 마이너 번호를 올린다: `1.0.4` → `1.0.5`.
   - **작은 핫픽스/버그 수정**은 하위 번호를 붙인다: `1.0.5` → `1.0.5.1`, `1.0.5.2` …
   - 기존 태그를 덮어쓰지 말고 **항상 새 태그**를 생성·푸시한다 (구버전과 혼동 방지, 추적성).

4. **브랜치 전략 (Branch Model)**
   - `main` = **안정 버전**. 반 친구들에게 공유하는 정식 빌드가 나오는 브랜치. 항상 실행 가능한 상태를 유지한다.
   - `dev` = **개발 버전**. 새 기능·실험은 여기서 진행한다.
   - 흐름: `dev`에서 작업·검증 → 안정되면 `main`에 병합 → `main`에서 버전 태그를 만들어 배포.

5. **배포 채널 & 자동 릴리스 (Release Channels)**
   - 태그를 push하면 GitHub Actions가 Windows exe(`DreamFarm.exe`)를 빌드해 릴리스로 올린다.
   - **안정 태그**(예: `v1.0.5`) → 정식 릴리스, GitHub에서 **Latest**로 표시(친구들이 받는 것).
   - **개발 태그**(예: `v1.0.5-dev.1`, `-beta`, `-rc` 포함) → **프리릴리스**로 표시(테스터용, Latest 아님).
